Task 2.2 WebSocket's have standardised technology to offer a fully asynchronous connection that has real time bi –direction communication between ser and browser. The nature of the connection being open for a period of time, means the connection is open for attacks. The first issue I noticed after implementing my WebSocket for the chat service, is that a client(user) can send any information through the WebSocket connection. Therefore, I decided to start by adding some client input validation. This can prevent attacks like OS, SQL, Blind SQL and many more. (Dabko Kovacic, 2021) To validate clientside input, I used an if statement to check if the user was a particular person, if the person doesn’t match that name, then nothing will be sent to the server. Users who have access to the system willo be notified and invited to use, of which they will receive notofocation through secure channels. My second security feature was server side validation. This means messages received on the client side should always be processed as data before being assigned to the DOM(document object model). I used a try/catch block along with JSON.parse to catch data types that was not a JSON string. I also used wss://, which is a much more safer protocol that ws:// which prevents things like man in the middle attacks. (Dabko Kovacic, 2021) I attempted to add rate limiting, but due this being my first time using Javascript, I defintelely need more time to implement this. It can be extemely imprtant as a security feature as it protects against bad bots, scrapers, clients who hog the connection without knowing or a malfunctioning client atha can lead to a DoS attack. Therefore limited the time of the client connection can be an adavantage.
